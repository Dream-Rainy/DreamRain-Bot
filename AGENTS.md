# Agent 行为规范

## 一、总体原则

- 所有对代码库的修改（包括文档更新）**必须先完整呈现计划，等待用户明确同意后才能执行**
- 文档是代码的镜像，只记录"对使用者有意义的变化"，不记录内部实现细节
- 宁可文档少，不要文档烂——不确定是否需要记录时，**默认不记录**

---

## 二、执行前必须 Plan

每次操作前，必须以如下结构向用户呈现完整计划，**未经确认不得执行任何步骤**：

```
## 📋 执行计划

### 将要修改的文件
- `src/xxx.kt` — [具体改动描述]
- `docs/yyy.md` — [具体改动描述]（如有）

### 文档变更评估
- 是否需要更新文档：是 / 否
- 原因：[为什么需要/不需要]
- 若需要，变更范围：[精确到节/行]

### 执行顺序
1. ...
2. ...

是否同意执行？(yes/no)
```

---

## 三、文档更新规则

### 3.1 必须更新文档的情形
仅在以下情况才更新 `README.md` 或相关文档：
- 新增/删除了公开 API 或公开函数签名
- 新增/删除了用户可见的功能或命令
- 修改了安装、构建、运行方式
- 修改了配置文件的字段或格式

### 3.2 不需要更新文档的情形
- 内部重构（逻辑不变，只改实现）
- Bug 修复（行为本应如此）
- 添加测试
- 修改注释或代码风格
- 性能优化（无 API 变更）

### 3.3 更新文档的方式
- **外科手术式修改**：只改动直接相关的段落或条目
- 禁止重写整个文档
- 禁止添加"此功能由 AI 生成"等无意义标注
- 禁止为每个函数自动生成文档块，除非用户明确要求

---

## 四、同意机制

- 用户回复 `yes` / `y` / `确认` / `同意` 才视为同意
- 用户回复 `no` / `n` / `取消` 视为拒绝，放弃本次操作
- 用户提出修改意见时，**重新生成 Plan** 后再次等待确认
- 禁止在用户未回复时自动推进任何步骤

---

## 五、完成后的汇报格式

操作完成后，以简洁格式汇报：

```
## ✅ 执行完成

### 已修改
- `src/xxx.kt`：[一句话描述]
- `docs/yyy.md`：[一句话描述]（如有）

### 未修改（及原因）
- `docs/zzz.md`：此次为内部重构，无公开 API 变更，无需更新
```

---

## 六、禁止行为

- ❌ 未经确认直接执行任何文件写入
- ❌ 因为"顺手"而更新与本次任务无关的文档
- ❌ 在文档中添加冗余的实现细节
- ❌ 将多个独立任务合并成一个 Plan 绕过逐步确认
